- purpose: iOS peer of the desktop duocb clipboard-sharing app — configure mode (a standing 47-char secret shared by all devices, per-device identity `<name>_<suffix>`, presence broadcast from the hub; the device list appears only behind the Join action, where the user picks the hosting peer) plus PIN quick pair (FFI roles `quick_host`/`quick_join`, identity-less; reachable from the pre-setup choice screen and the hub). Quick pair's channel menu selects `nostr_lan` (default, the desktop "P" preset — the relays carry the rendezvous on iOS) or `lan` (the desktop "L" preset — no third-party server: a Bonjour/DNS-SD service through the system mDNSResponder daemon, exempt from Apple's restricted multicast entitlement, with the join dialing the resolved direct addresses; Info.plist lists `_duocb-pin._udp` under NSBonjourServices and joining triggers the Local Network prompt). Both devices must pick the same channel. The nostr-only PIN preset and manual mode remain desktop-only
- strict no backward compatibility
- make changes on the sibling project ../duocb (the Rust core + FFI used by this app) if needed
- the Rust artifact (libduocb.xcframework) is delivered via a local Swift package `Packages/Duocb/Package.swift` (referenced by local path in project.yml); its binary target downloads the pinned release zip by URL+checksum (SPM auto-exposes the embedded duocb.h to the bridging header, so no vendor/ dir and no HEADER_SEARCH_PATHS). Bump to a new release with `scripts/bump-xcframework.sh <tag>`. For FFI dev against a local Rust build, run `./build-ios.sh` in ../duocb then set `DUOCB_LOCAL_XCFRAMEWORK=1` for both `xcodegen generate` and `xcodebuild` — SPM forbids binary paths outside the package root, so the sibling's dist/ios is reached via the committed symlink `Packages/Duocb/local/libduocb.xcframework`
- minimum iOS 17 (SwiftUI + @Observable; no WebView, no Network Extension, no background modes — v1 is foreground-first)
- regenerate the project from project.yml after file additions/renames (`xcodegen generate`)
- build/verify with an arm64 simulator pinned, e.g. `xcodebuild -project Duocb.xcodeproj -scheme DuocbApp -destination 'platform=iOS Simulator,name=iPhone 17' build` — libduocb.xcframework is arm64-only, so a generic `-sdk iphonesimulator` destination picks x86_64 and fails to link
- persistence policy mirrors the desktop wizard: the secret lives in the Keychain (TokenStore, saved the moment it is generated/imported), the device name in UserDefaults (saved the moment it is confirmed), and the permanent 8-char identity suffix in the Keychain (SuffixStore, minted once via duocb_generate_suffix and never regenerated — it survives Clear secret). Editing a wizard field alone never persists
- runtime instances (SessionController): at most one FFI handle at a time — a "hub" instance (presence + peer list) runs while the hub screen is visible, and is stopped and replaced by a "start"/"join" instance when the user hosts or taps a hosting peer; Stop tears the session down and the hub restarts on reappear
